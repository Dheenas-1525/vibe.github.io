Phase 1.1 · Database

🍃 MongoDB Atlas

MongoDB Atlas is a free cloud database. ViBe stores everything here — users, courses, quizzes, progress. Setup takes 5 minutes.

💡
What is a database?Think of it as a filing cabinet in the cloud. Every time a teacher creates a course or a student takes a quiz, that data gets saved here automatically.
1

Create a MongoDB Atlas account

Go to mongodb.com/atlas → click Try Free → sign up with your email → verify your email.

2

Create a free cluster

Click Build a Database → choose M0 Free → pick any cloud provider and the region closest to your server → name it vibe-cluster → click Create Deployment.

3

Create a database user

A popup appears. Enter a username (e.g. vibe_user) and click Autogenerate Secure Password. Copy the password — you need it in the next step. Click Create Database User.

4

Allow network access

Click Allow Access from Anywhere and enter 0.0.0.0/0. This lets your server connect to the database. Click Finish and Close.

5

Get your connection string (DB_URL)

Click Connect on your cluster → choose Drivers → select Node.js. Copy the connection string. Replace <password> with your actual password. This is your DB_URL.

Your DB_URL looks like this
mongodb+srv://vibe_user:YOUR_PASSWORD@vibe-cluster.abc123.mongodb.net/?retryWrites=true&w=majority&appName=vibe-cluster
✅
No tables to createViBe automatically creates all database collections when it first starts up. You do not need to set up any tables or schemas manually.
Phase 1.2 · Authentication

🔥 Firebase

Firebase handles user login and sign-up. You will get 7 configuration values from the Firebase Console.

1

Create a Firebase project

Go to console.firebase.google.com → click Add Project → name it (e.g. vibe-myschool) → skip Analytics → click Create Project.

2

Enable Email/Password sign-in

Left sidebar → Authentication → Get Started → Email/Password → toggle Enable → Save.

3

Register a Web App and copy config

Project home → click </> icon → nickname it vibe-web → click Register App. You will see a config block with 7 values. Copy all of them.

4

Add your domain to authorized domains

Authentication → Settings → Authorized Domains → Add Domain → enter your server's IP or domain name.

5

Generate a Service Account Key

Click the gear icon → Project Settings → Service accounts tab → Generate new private key → confirm → a JSON file downloads. Save it as firebase-service-account.json in the vibe/ project root. This is required for the backend to verify user tokens.

Phase 1.4 · Security

🛡️ Google reCAPTCHA

Prevents bots from creating fake accounts. You get two keys — one for the frontend, one for the backend.

1

Go to the reCAPTCHA admin console

Visit google.com/recaptcha/admin/create and sign in with a Google account.

2

Register your site

Label: any name. Type: reCAPTCHA v2 → "I'm not a robot" Checkbox. Domains: add your server IP or domain (also add localhost for testing). Accept terms → Submit.

3

Copy both keys

Site Key (public) → goes in frontend/.env as VITE_RECAPTCHA_SITE_KEY
Secret Key (private) → goes in backend.env as RECAPTCHA_SECRET_KEY

Phase 2.2 · Tools

🐳 Install Docker & Git

Docker runs ViBe in isolated containers. Git downloads the source code.

bash — Ubuntu / Debian
# Update package list
sudo apt update

# Install Git
sudo apt install git -y

# Install Docker dependencies
sudo apt install ca-certificates curl gnupg -y
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

# Add Docker repository
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Install Docker
sudo apt update && sudo apt install docker-ce docker-ce-cli containerd.io docker-compose-plugin -y

# Allow running Docker without sudo
sudo usermod -aG docker $USER && newgrp docker

# Verify
docker --version && docker compose version
💡
Download and install Docker Desktop from docker.com/products/docker-desktop — it includes Docker Compose. Git comes pre-installed on macOS (run git --version in Terminal to confirm).
PowerShell (as Administrator)
# Enable WSL2

Phase 4.1 · Launch

🚀 Launch with Docker

Three commands to pull, build, and start everything.

1

Pull the backend Docker image

The backend is pre-built on Docker Hub. This downloads it to your server.

bash
docker pull vicharanashala/vibe-backend:staging
2

Build the frontend image

The frontend is built from your source code so your Firebase and API URL values get baked in. This takes 3–10 minutes.

bash
docker compose build frontend
3

Start everything

The -d flag runs containers in the background.

bash
docker compose up -d
# Check what's running
docker compose ps

Phase 4.3 · Recommended

🔒 HTTPS with a Domain

For production use, set up a real domain and SSL certificate. Free with Let's Encrypt.

1

Install Nginx and Certbot

bash
sudo apt install nginx certbot python3-certbot-nginx -y
2

Create Nginx config

Replace yourdomain.com with your actual domain.

nginx
server {
    listen 80;
    server_name yourdomain.com;

    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
    }
    location /api {
        proxy_pass http://localhost:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
3

Enable and get SSL certificate

bash
sudo ln -s /etc/nginx/sites-available/vibe /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
sudo certbot --nginx -d yourdomain.com
✅
After HTTPS is set upUpdate APP_URL, APP_ORIGINS, FRONTEND_URL in backend.env and VITE_BASE_URL in frontend/.env to use https://. Then rebuild: docker compose build frontend && docker compose up -d

It has no AI inside — it is purely a translator. Enable it by uncommenting the litellm service in docker-compose.yml and creating litellm_config.yaml from the example in self-hosting/.

What to do when a new version of ViBe is released.

Operations · Help

🔧 Troubleshooting

Backend container keeps restarting

Run docker compose logs backend and look at the last few lines. Most common causes:

→ MongoNetworkError — wrong DB_URL or IP not whitelisted in Atlas
→ Invalid API Key — wrong ANTHROPIC_CRED
→ EADDRINUSE — port 8080 already in use on your server

Sign-up fails / login page shows error

→ Firebase Email/Password auth not enabled — check Firebase Console → Authentication → Sign-in providers
→ Your domain/IP not in Firebase Authorized Domains — add it in Firebase → Authentication → Settings
→ Wrong VITE_FIREBASE_* values — rebuild frontend after fixing: docker compose build frontend && docker compose up -d

API not reachable / blank page in app

→ Test: curl http://localhost:8080/health — if it fails, backend is down
→ VITE_BASE_URL must be reachable from the user's browser (not just from the server)
→ APP_ORIGINS in backend.env must include your frontend URL
→ Rebuild frontend after any env change: docker compose build frontend

Port already in use

bash
sudo lsof -i :3000   # find what's using port 3000
sudo kill -9 <PID>

Frontend env changes have no effect

Frontend env variables are baked in at build time. After any change to frontend/.env you must rebuild:

bash
docker compose build frontend && docker compose up -d frontend

Video pipeline: transcript / questions show "Load failed"

After the AI finishes a stage, clicking the transcript or question preview shows "Load failed". This was caused by the browser trying to fetch files from the internal Docker hostname (minio:9000) which is not reachable outside Docker. Fixed in the current codebase — the backend now rewrites internal URLs to /gcs/... and Nginx proxies those requests to MinIO.

→ If you see this, rebuild the backend and frontend: docker compose build backend frontend && docker compose up -d
→ Confirm GCS_INTERNAL_URL=http://minio:9000 and GCS_EXTERNAL_URL=/gcs are set in backend.env
→ Confirm the /gcs/ location block is present in frontend/nginx.conf

Video pipeline: MinIO upload fails / storage unavailable

If stages fail with storage errors, MinIO is likely misconfigured or not running.

→ Check MinIO is running: docker compose ps | grep minio
→ Check credentials match: MINIO_ACCESS_KEY / MINIO_SECRET_KEY in vibe-ai/.env must equal MINIO_ROOT_USER / MINIO_ROOT_PASSWORD in docker-compose.yml
→ The bucket vibe-aiserver-data is auto-created on first start — no manual setup needed
→ View files via MinIO web UI (SSH tunnel): ssh -L 9001:localhost:9001 user@server → open http://localhost:9001

Video pipeline: question generation produces empty results

The job completes but 0 questions are saved. This was caused by the LLM being asked to generate all questions in one call — the JSON response exceeded the model's token limit and was silently truncated. This is fixed in the current codebase — questions are now generated one at a time.

→ Watch live progress: docker logs vibe-aiserver --follow
→ You should see: Generated SOL question 1/10 for segment ...
→ If errors appear: check LM Studio is loaded and responding: curl http://localhost:1234/v1/models
→ If you rebuilt the container and questions are still empty, confirm you're running the rebuilt image: docker compose build vibe-aiserver && docker compose up -d vibe-aiserver

Video pipeline: question generation loops / job keeps restarting

Questions generate partially (e.g. 1–5 succeed) then stop with "No models loaded" errors for the rest. The backend re-triggers the job and the cycle repeats. Root cause: LM Studio's auto-unload feature drops the model during the idle gap between question calls.

→ Fix: Open LM Studio → Settings (gear icon, bottom-left) → Performance tab → set "Automatically unload model after X minutes of inactivity" to Never
→ Confirm the model stays loaded during the full run: curl http://localhost:1234/v1/models
→ Verify in logs — all 10 lines should appear without errors: docker logs vibe-aiserver --follow

Publishing learning module fails / "Uploaded to course" error

After reviewing questions, clicking Publish to Course fails with a 500 error. Root cause: MongoDB write conflict (error code 112) during the long publish transaction — the existing retry logic only caught TransientTransactionError-labelled errors. Fixed in the current codebase — BaseService.ts now also retries on write conflict code 112 with backoff.

→ If you see this error, rebuild and restart the backend: docker compose build backend && docker compose up -d backend
→ Then try publishing again — the transaction will retry automatically on conflict
→ Confirm in logs: docker logs vibe-backend --follow — no more Write conflict during plan execution errors

Video pipeline: no logs from question generation in docker logs

If docker logs vibe-aiserver stops at "Starting question generation task" with nothing after — the container was built without PYTHONUNBUFFERED=1. Background thread output was buffered and never flushed. Fixed in the current Dockerfile.

bash — rebuild to get the fix
docker compose build vibe-aiserver
docker compose up -d vibe-aiserver

Video pipeline: job stuck after container restart

In-memory task state is lost on restart. If the backend cannot abort a job, it stays permanently at RUNNING. The abort endpoint now returns HTTP 200 even when no task is found — allowing the backend to mark the job ABORTED so it can be retried.

→ In the dashboard: click Abort → wait for status to change to ABORTED → click Rerun
